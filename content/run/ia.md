# Information Architecture — Feedback→Roadmap

## Главный экран

**Имя экрана:** Inbox (Входящие)

**Главное действие:** Разобрать входящий фидбэк

**Обоснование выбора:**

Критерий 1 (приоритет): Inbox закрывает JTBD-1 («стянуть фидбэк из чата, не теряя») — это
первый и ядерный JTBD по research.md. Фичи #1, #2, #3 из таблицы must (строки с highest
частотой использования) все замыкаются именно здесь: подключённая интеграция гонит данные
сюда, а захват и отображение единого бэклога — это Inbox.

Критерий 2 (shortcuts): Из Backlog, Roadmap и Settings пользователь возвращается в Inbox
при каждой новой партии фидбэка. Это единственный экран с регулярным pull (PM открывает
инструмент именно чтобы проверить входящие).

Критерий 3 (время): Типичный сценарий «понедельничный разбор» (research.md §5) проводится
на Inbox — просмотр, фильтрация шума, принятие/отклонение. Это наиболее трудоёмкий этап.

Победа по всем трём критериям без конфликта → Inbox.

## User Flow — главный сценарий

Сценарий: «Понедельничный разбор» — PM разбирает накопившийся за неделю фидбэк из Telegram
и переводит приоритетные пункты в роадмап.

| # | Экран | Действие пользователя | Результат / Следующий экран |
|---|-------|-----------------------|-----------------------------|
| 1 | Login | Вводит email + пароль (или OAuth) | Inbox |
| 2 | Inbox | Просматривает список входящих элементов из Telegram | Inbox — список элементов с автором, датой, источником |
| 3 | Inbox | Отмечает нерелевантные сообщения как «шум» / отклоняет | Элементы скрыты из активного вида; счётчик уменьшается |
| 4 | Inbox | Нажимает «Принять» на релевантном элементе | Элемент перемещается в Backlog; Inbox обновляется |
| 5 | Backlog — Группы | Просматривает авто-сгруппированные темы | Backlog с группами, счётчиком вхождений на группу |
| 6 | Backlog — Группы | Перетаскивает/привязывает элемент к существующей группе или создаёт новую | Группа обновлена; счётчик частоты вырос |
| 7 | Backlog — Приоритизация | Выставляет Value (1-5) и Effort (1-5) для группы/темы | Система пересчитывает ранг (Value/Effort); список переупорядочивается |
| 8 | Backlog — Приоритизация | Вручную перетаскивает пункт вверх/вниз если нужна корректировка | Ручной ранг зафиксирован; значок «скорректировано вручную» |
| 9 | Backlog — Приоритизация | Нажимает «Добавить в роадмап» на топ-пунктах | Пункты появляются в Roadmap в колонке «To Do» |
| 10 | Roadmap — Kanban | Просматривает доску; перетаскивает пункт в колонку «In Progress» | Статус пункта обновлён; команда видит изменение |


## Edge Cases

### Edge Case 1: Empty State

**Когда возникает:** Пользователь зарегистрировался, но ещё не подключил Telegram-интеграцию,
или подключил, но фидбэк ещё не поступал.

**Какой экран:** Inbox.

**Что показывается:**
- Иллюстрация/иконка пустого inbox.
- Заголовок: «Пока нет фидбэка».
- Подзаголовок: «Подключите Telegram-бота, чтобы фидбэк из чатов автоматически попадал сюда».
- Кнопка CTA: «Подключить Telegram» → ведёт в Settings → Интеграции.
- Если интеграция уже подключена, но фидбэка нет: «Интеграция активна. Перешлите любое
  сообщение вашему боту — оно появится здесь».

**Куда зовём:** Settings → Интеграции (если бот не подключён) или бот-инструкция (если подключён).

---

### Edge Case 2: Error State

**Наиболее вероятные ошибки:**

| Тип ошибки | Где возникает | Что показываем | Как восстанавливаемся |
|---|---|---|---|
| Telegram API недоступен / токен истёк | Inbox (нет новых элементов при ожидаемом поступлении) | Баннер: «Не удаётся получить данные из Telegram. Проверьте интеграцию.» + кнопка «Настройки» | Переход в Settings → Интеграции → повторная авторизация бота |
| Ошибка авто-группировки | Backlog — Группы | Элемент в разделе «Без группы» с иконкой предупреждения | PM вручную привязывает к группе |
| Ошибка сети при сохранении приоритета | Backlog — Приоритизация | Toast: «Не удалось сохранить. Проверьте соединение.» + кнопка «Повторить» | Retry; данные не теряются (оптимистичный UI с откатом) |
| Ошибка OAuth при первичном подключении | Settings — Интеграции | Inline-ошибка под шагом авторизации с кодом и ссылкой на поддержку | Повторить OAuth-поток |

---

### Edge Case 3: First-Time vs Returning User

**Первый запуск (onboarding):**
- После регистрации — короткий onboarding wizard (3 шага, не более):
  1. «Подключите Telegram-бота» — OAuth + инструкция по форварду.
  2. «Перешлите первое сообщение» — живая демонстрация захвата (или sample item если
     пользователь пропускает шаг).
  3. «Вот ваш первый бэклог» — показывает Inbox с одним элементом, объясняет действия.
- Нужен sample data: если пользователь пропустил шаг 2, показываем 2-3 синтетических
  «примера фидбэка» с пометкой «Пример — замените реальным фидбэком».
- После wizard → Inbox с CTA «Разберите первую партию».

**Возвращающийся пользователь:**
- Открывает продукт → сразу Inbox с новыми элементами (без wizard).
- Если новых элементов нет — Inbox с last-seen датой последней синхронизации и кнопкой
  «Обновить».
- Если отсутствовал >7 дней и накопилось >20 элементов — баннер: «За неделю пришло X
  элементов — начните разбор».


## Связь с JTBD из research.md

| JTBD | Где закрывается (экран) |
|------|--------------------------|
| JTBD-1 — стянуть фидбэк из чата, не теряя | Inbox + Settings → Интеграции (настройка бота) |
| JTBD-2 — не разбирать сотни сообщений руками | Backlog → Группы / Темы (авто-группировка + мёрдж) |
| JTBD-3 — понять, за что браться | Backlog → Приоритизация + Roadmap → Kanban |
| JTBD-4 — привязать фидбэк к теме и видеть «вес» | Backlog → Группы (привязка + счётчик частоты) |
| JTBD-5 — показать команде/юзерам, почему так | Roadmap → Kanban (прозрачный статус + приоритет) |
| JTBD-6 — отделить фидбэк от болтовни | Inbox (ручное принятие/отклонение) + Settings → Правила захвата |


## Открытые вопросы и допущения

### Допущения (приняты для разблокирования IA, требуют подтверждения заказчика)

**Допущение D1 — Статусы роадмапа:**
Принят минимальный набор из 3 колонок: «To Do» / «In Progress» / «Done».
Публичность роадмапа для пользователей — вынесена в v2 (фича #9 research.md).
Основание: brief.md явно помечает «что входит в базовый роадмап» как открытый вопрос;
минимальный Kanban — наименьшее разумное допущение, не добавляющее сложности к MVP.
*Требует подтверждения заказчика.*

**Допущение D2 — Один workspace на команду:**
MVP проектируется под одну команду / один workspace. Мультикомандная архитектура (несколько
продуктов в одном аккаунте) — вне скоупа MVP согласно brief.md («малая команда/инди»).

**Допущение D3 — Авторизация:**
Стандартный email + пароль / Google OAuth. SSO и командные приглашения — минимальны
(Settings → Рабочее пространство и участники содержит базовое управление членами команды).
